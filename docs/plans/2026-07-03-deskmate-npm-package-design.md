# Deskmate as an importable npm package — design

> **Goal:** Turn Deskmate from a *clone-this-repo-and-deploy* template into an
> **importable npm package configured by a single `deskmate.config.ts`**. A consumer
> runs `npm i @deskmate/core`, writes a config, and gets a deployable Eve app — instead
> of forking the repo.
>
> **Date:** 2026-07-03 · **Repo:** this one (`deskmate`) becomes the package monorepo ·
> **First real consumer:** AddMeIn, as a config-only follow-on in `Addmein-ai/addmein`
> (out of scope for this change).
>
> **Process:** shaped with the superpowers brainstorming skill. "Implement
> `/using-superpowers`" meant *use the superpowers methodology* (brainstorm → write-plan
> → execute), not build a skills feature.

---

## The one constraint that shapes everything

Eve discovers agents **by filesystem convention**, not programmatically. A subagent is a
subagent *only because it lives at `agent/subagents/<id>/`*; the compiler walks those
directories at build time. There is no `defineAgent({ subagents: [...] })` API.

So "import Deskmate and configure it" **cannot** mean the package *is* the agent tree.
The directories must physically exist when `eve build` runs. The design bridges that gap
with a generator: `deskmate.config.ts` → generated `agent/**` shims (see [Sync](#the-cli--what-sync-generates)).

---

## Decisions locked during design

- **Monorepo, fully OSS (Apache-2.0).** This repo becomes a pnpm workspace. Every package
  in it is open source. The core *is* the product's identity ("open-source, self-hostable
  AI coworkers") — closing it would collapse the positioning.
- **Commercial layer stays external and deferred.** The hosted layer (multi-tenant control
  plane, per-tenant secret vault, onboarding wizard, dashboard, billing, provisioning —
  already scoped *out* in the README) is **not** built and does **not** live in this repo.
  When it exists, it becomes a separate **private** repo that depends on `@deskmate/core`
  like any other consumer. We design the seam (connections resolve auth per-call,
  deskmates are data-driven); we do not build the wall. No mixed-license monorepo.
- **Naming:** keep **deskmate** (not "employee"). Top-level config helper is **`defineTeam`**
  (a team of deskmates); per-member helper is **`defineDeskmate`**.
- **No engine guardrails, no shared-instruction stitching (YAGNI).** A deskmate's
  `instructions.md` is the **authored role file, verbatim**. The engine is unopinionated
  about instruction content. A consumer who wants a cross-cutting boundary (e.g. AddMeIn's
  privacy rule) writes it in its own role files. Revisit a `rules: [...]` mechanism only
  if write-once-apply-to-all becomes a real pain across a large roster.
- **Generated shims are committed** and regenerated by `deskmate sync` as a prebuild step
  — the successor to today's auto-generated `agent/lib/deskmates.ts`.
- **Generic third-party readers are catalog examples, not engine APIs.** Fly/Twilio/Stripe/
  Postgres/GitHub/Langfuse readers ship as copy-and-adapt examples in the catalog so the
  engine's public API stays small.

## Packages (monorepo layout)

```
deskmate/                      ← this repo (pnpm workspace, Apache-2.0)
  packages/
    core/     @deskmate/core   Engine. Small, stable, the only import with a contract:
                               • defineTeam / defineDeskmate
                               • front-desk router + convene loop
                               • deskmate_says, identity, avatars, channel-routes
                               • skill loading + generic MCP connection helpers
    cli/      deskmate         The `deskmate` command (bundles the catalog content):
                               • add/remove <role>   scaffold editable role into your repo
                               • sync                config → generated agent/ tree
                               • init, mcp-add
    catalog/                   Copy-and-adapt content (NOT a published package, NO contract):
                               • roles/<id>/     the 5 current roles (instructions, skill,
                                                 read tool w/ seed, starter config entry)
                               • connections/    reader EXAMPLES: posthog, github, postgres,
                                                 langfuse, fly, twilio, stripe (seed data)
  examples/
    starter/                   A real, deployable Eve app: imports @deskmate/core, has a
                               deskmate.config.ts (product_analyst + devops), runs
                               `deskmate sync`, deploys to Vercel. Dogfood + README quickstart.
```

**How today's tree maps in** (relocate, don't rewrite):
- `library/deskmates/` → `packages/catalog/roles/`
- `scripts/deskmate.ts` (+ `scripts/lib/*`) → `packages/cli/`
- `agent/lib/{convene,deskmate-identity,deskmate-avatars,channel-routes}.ts`,
  `agent/tools/deskmate_says.ts`, `agent/channels/*` → `packages/core/`
- current root `agent/` app → `examples/starter/agent/` (now **generated** by `sync`)

## The config API & the three file kinds

A consumer has exactly three kinds of files:

| Kind | Files | Owner |
|---|---|---|
| **Config** | `deskmate.config.ts` | You write it — single source of truth for the roster |
| **Authored** | `roles/<id>/instructions.md`, `tools/*.ts`, `connections/*.ts` | Seeded by `deskmate add`, then **you edit + own** |
| **Generated** | `agent/**` | `deskmate sync` owns it — committed, "do not edit" |

```ts title="deskmate.config.ts"
import { defineTeam } from "@deskmate/core";

export default defineTeam({
  model: "anthropic/claude-opus-4.8",     // team default; per-deskmate can override
  frontDesk: { maxTurns: 6 },             // today's DESKMATE_MAX_TURNS

  connections: {                          // named, read-only, referenced by deskmates
    posthog:    { kind: "mcp", env: "POSTHOG" },            // → POSTHOG_MCP_URL/_TOKEN
    github:     { kind: "mcp", env: "GITHUB", repo: "Addmein-ai/addmein" },
    postgres:   { kind: "mcp", env: "POSTGRES" },
    "fly-logs": { kind: "tool", from: "./connections/fly", apps: ["addmein-web"] },
  },

  deskmates: {
    product_analyst: {
      role: "product_analyst",            // catalog role → seeds instructions/skill/tool
      emoji: "📊", displayName: "Product Analyst", summary: "…",
      reads: ["posthog", "postgres", "github"],   // connection names
      skill: "product-analyst",
      // instructions + tools default to the copied role files; override per-key if needed
    },
    devops: { role: "devops", emoji: "🔧", displayName: "DevOps Engineer",
              summary: "…", reads: ["fly-logs", "github"] },
  },

  channels: { C0123INCIDENTS: { deskmate: "devops", lock: true } },
});
```

**Identity (`emoji`/`displayName`/`summary`) lives in the config**, so the whole roster
reads at a glance in one file. **The spectrum:** an all-stock roster is just config + `.env`
(nothing authored); customization means editing the copied `roles/<id>/instructions.md` or
dropping a local `tools/*.ts` and referencing it.

## The CLI & what `sync` generates

Two commands do the work:

- **`deskmate add <role>`** — copies a catalog role into your **editable** space
  (`roles/<id>/instructions.md`, a read tool with seed data, any connection example) and
  appends a starter entry to `deskmate.config.ts`. One-time scaffold; you own the result.
- **`deskmate sync`** — reads `deskmate.config.ts` and (re)generates the whole `agent/**`
  tree. Idempotent, committed, runs as a prebuild step: `"build": "deskmate sync && eve build"`.

**What `sync` writes** for the config above:

```
agent/
  agent.ts                     ← front-desk model + config (from defineTeam)
  instructions.md              ← front-desk router (from @deskmate/core) + channel routes
  channels/…                   ← Slack + eve channels (from @deskmate/core)
  lib/deskmates.ts             ← roster registry (as today)
  subagents/
    product_analyst/
      agent.ts                 ← defineDeskmate(team, "product_analyst")  (description from config)
      instructions.md          ← your roles/product_analyst/instructions.md, VERBATIM
      tools/get_metric_snapshot.ts   ← re-export shim → your tools/…
      connections/posthog.ts   ← generated from config.connections.posthog (env-driven)
      connections/postgres.ts
      connections/github.ts
    devops/ …
  # also refreshes .env.example from every connection's env keys
```

Every generated file carries `// GENERATED by deskmate sync — edit deskmate.config.ts`.
`deskmate remove <id>` + re-`sync` cleans the tree.

**The one rule that keeps it sane:** you edit **config + authored files**; you never edit
`agent/`. `sync` owns `agent/` completely and can rebuild it from scratch at any time.

Example generated shim:

```ts title="agent/subagents/product_analyst/agent.ts"
// GENERATED by deskmate sync — edit deskmate.config.ts instead
import { defineDeskmate } from "@deskmate/core";
import team from "../../../deskmate.config";
export default defineDeskmate(team, "product_analyst");
```

## Migration (relocate, don't rewrite)

Each step keeps the existing test suite green:

1. pnpm workspace + empty `packages/{core,cli,catalog}` + `examples/starter`.
2. `agent/lib/*`, `deskmate_says`, `channels/*` → `packages/core` as exports (pure/tested
   functions intact).
3. `library/deskmates/*` → `packages/catalog/roles/*`; lift shared readers →
   `packages/catalog/connections/*`.
4. `scripts/deskmate.ts` → `packages/cli`, retargeted: `add` = scaffold editable,
   `sync` = generate `agent/` from config.
5. Current root `agent/` → `examples/starter/` driven by a `deskmate.config.ts` that
   reproduces today's active roster; delete hand-authored subagents in favor of generated ones.
6. README → the new install / configure / sync flow.

## Testing

- **The existing 14 tests move with their code and stay green** — the safety net proving
  the migration changed no behavior.
- **New tests:**
  - `deskmate sync` generator — config → expected `agent/` tree; idempotent; `remove` cleans up.
  - `defineTeam` config validation (zod schema).
  - Smoke test: `examples/starter` runs `deskmate sync && eve build` clean — the
    end-to-end proof that import → configure → sync → build works.
- **CI:** workspace-wide typecheck + test, plus build the example.

## AddMeIn (out of scope here; the first real consumer)

AddMeIn's roster ([2026-07-02 design](./2026-07-02-addmein-roster-design.md)) becomes a
**config-only** consumer, living at **`apps/deskmate/` inside `Addmein-ai/addmein`** (matches
the `apps/` convention; deploys to its own Vercel project). It depends on `@deskmate/core`,
copies the catalog readers it needs into `apps/deskmate/{tools,connections}/` and adapts them,
and owns its `deskmate.config.ts` (the 9 roles, GitHub repo, Fly apps, channel routes, and its
own privacy-rule wording). **No AddMeIn-specific code lives in the OSS monorepo.**

## Out of scope (YAGNI)

- The commercial/hosted layer (separate private repo, later).
- Engine guardrails / a `rules` shared-instruction mechanism (revisit with real need).
- Gitignored build-artifact shims (we commit them).
- A separately published `@deskmate/catalog` package (catalog is copy-from content the CLI bundles).
- Runtime skill-discovery beyond what Eve already provides.
- The full AddMeIn 9-role migration (a config-only follow-on).
