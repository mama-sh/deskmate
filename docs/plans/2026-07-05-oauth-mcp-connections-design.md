# OAuth MCP connections (app-scoped Vercel Connect)

**Date:** 2026-07-05
**Status:** Approved design â€” ready for implementation planning

## Problem

Deskmate's MCP connection model only understands static tokens. A connection is
declared as `{ kind: "mcp", env: "<PREFIX>" }`, which maps to a
`<PREFIX>_MCP_URL` + `<PREFIX>_MCP_TOKEN` pair, and the scaffolder
(`deskmate mcp-add`) emits an `auth: { getToken }` connection file.

Several target MCP servers are OAuth-only and cannot be driven by a static
token â€” notably **Vercel** (`mcp.vercel.com`) and **Neon** (hosted MCP). There
is no way to express such a connection in `deskmate.config.ts`, no scaffolding
for it, and the generated `.env.example` / stub files assume the token model.

## Key finding: eve already does the OAuth

Eve supports OAuth-backed MCP servers first-class via **Vercel Connect**. The
`connect()` helper from `@vercel/connect/eve` owns the browser consent flow,
encrypted token storage, refresh, and project access, so credentials never enter
model context. `@vercel/connect@0.3.2` is **already** a dependency of
`@deskmate/core` and `examples/starter`.

So deskmate does not implement OAuth. The gap is entirely in deskmate's own
layer â€” config model, scaffolder, `.env.example`, and stub renderer â€” none of
which can currently express a Connect connection.

Connect offers two principal scopes:

- **App-scoped** â€” `connect({ connector, principalType: "app" })`: the agent
  acts as *itself*, non-interactive, one shared app token.
- **User-scoped** â€” `connect("<uid>")`: per-user consent, interactive
  park/resume, requires an authenticated user principal.

## Decisions

1. **App-scoped only.** Deskmates are proactive/autonomous â€” they act as the bot
   in Slack channels, with no signed-in human whose principal user-scoped Connect
   would require. App-scoped is non-interactive and the correct fit. User-scoped
   is intentionally out of scope (YAGNI; can be added later if a real case
   appears).

2. **Config shape: additive optional `connect` field** (chosen over a
   discriminated `auth` sub-object or a new `kind: "mcp-oauth"`). Keeps
   `kind: "mcp"`, is backward compatible with every existing config, and reduces
   the auth-mode branch everywhere to one predicate ("is `connect` set?").

3. **CLI: `mcp-add` oauth branch + a new `deskmate connect` command.** `mcp-add`
   scaffolds the connection file + config entry; `deskmate connect` shells out to
   the Vercel CLI to provision the connector â€” consistent with how
   `deskmate deploy` already wraps `vercel build` / `vercel deploy`.

4. **Scope: reusable capability only.** Actually wiring addmein's Vercel + Neon
   connectors (running `vercel connect create/attach` against the real account)
   is a follow-up that consumes this capability, not part of this branch.

## Design

### 1. Config model â€” `packages/core/src/config.ts`

```ts
const ConnectionConfig = z.object({
  kind: z.literal("mcp"),
  env: z.string().optional(),      // token model â†’ <ENV>_MCP_URL/_TOKEN
  connect: z.string().optional(),  // oauth model â†’ app-scoped Vercel Connect connector UID
}).refine((c) => !(c.env && c.connect), {
  message: "a connection uses either `env` (token) or `connect` (oauth), not both",
});
```

`connect` holds the connector UID (e.g. `"vercel/deskmate"`). No new `kind`. No
new `defineTeam` identifier rule â€” the UID is not used as a filesystem path
segment. A connection is OAuth iff `connect` is set, token iff `env` is set.

```ts
connections: {
  sentry: { kind: "mcp", env: "SENTRY" },              // token (unchanged)
  vercel: { kind: "mcp", connect: "vercel/deskmate" }, // oauth
}
```

### 2. Scaffolding â€” `mcp-add.ts` + `lib/mcp-template.ts`

`deskmate mcp-add <name>` gains an auth-mode branch (first prompt `token` /
`oauth`, default `token`).

- **token** path: unchanged.
- **oauth** path: prompts connector UID (default `<name>/deskmate`), MCP URL
  (e.g. `https://mcp.vercel.com`), description, read tools. Writes:

```ts
import { connect } from "@vercel/connect/eve";
import { defineMcpClientConnection } from "eve/connections";

// Generated by `deskmate mcp-add` (oauth). App-scoped Vercel Connect.
// Provision the connector with `deskmate connect <name>`.
export default defineMcpClientConnection({
  url: "https://mcp.vercel.com",
  description: "â€¦",
  auth: connect({ connector: "vercel/deskmate", principalType: "app" }),
  tools: { allow: ["â€¦"] },
});
```

Config entry appended: `{ kind: "mcp", connect: "vercel/deskmate" }`.
`renderMcpConnection` splits into the existing token renderer plus a new connect
renderer.

### 3. `deskmate connect <name>` â€” new `connect.ts`, wired into `cli.ts`

Reads the config, resolves `connections[name].connect` (the UID) and the
connection's URL from the scaffolded `connections/<name>.ts` (or
`roles/*/connections/<name>.ts`). Then shells out using the same
`spawn`-with-clear-ENOENT DI pattern `deploy.ts` uses:

```
vercel connect create <service> --name <connector-name>   # service = URL host, name = UID after "/"
vercel connect attach <uid> --yes
vercel env pull
```

- Requires the Vercel CLI installed + authenticated (documented, like
  `deskmate deploy`). A spawn ENOENT emits a clear "install/authenticate the
  Vercel CLI" message.
- Surfaces the CLI's "already exists" rather than failing hard, so re-running is
  safe.
- Open sub-decision, settled in the plan: deriving `service` (Connect service
  identifier) and `connector-name` from the URL host + UID. Fallback if
  host-derivation is unreliable: store `service` explicitly in the config entry.

### 4. Renderer touch-ups â€” `sync/render.ts`

- `renderEnvExample`: the existing filter
  `.filter(([, c]) => c.kind === "mcp" && !!c.env)` already excludes OAuth
  connections from the URL/TOKEN pairs. Add an "OAuth connections (Vercel
  Connect)" comment block listing them by name + connector UID, pointing at
  `deskmate connect`.
- `renderStubConnection`: when a `read` connection has `connect` set but no
  authored file, emit a **connect-flavored** stub
  (`auth: connect({ connector, principalType: "app" })`, TODO url) instead of the
  token stub, so `eve build` stays green and the gap is obvious.

### 5. Starter + docs

- `examples/starter/deskmate.config.ts`: add a commented OAuth connection example
  next to the token ones.
- README / AGENTS: short "OAuth (Vercel Connect) connections" section â€”
  app-scoped rationale, the `mcp-add` oauth flow, `deskmate connect`, and the
  Vercel-CLI-auth prerequisite. `@vercel/connect` is already a dependency, so no
  new install.

### 6. Testing

- `mcp-template.test.ts`: connect-renderer output (imports `connect`, app-scoped,
  correct UID / url / tools).
- `mcp-add.test.ts`: oauth prompt branch writes the connect file +
  `{ kind: "mcp", connect }` config entry; bad-input / mutual-exclusion paths.
- `config.test.ts`: `connect` parses; `env` + `connect` together is rejected by
  the refine.
- `render.test.ts`: OAuth connection omitted from `.env.example` URL/TOKEN pairs
  but listed in the comment block; connect stub shape.
- New `connect.test.ts`: the command resolves the UID, derives service/name, and
  issues the right `vercel` argv against an injected `run` (same DI seam
  `deploy.ts` uses) â€” no real network.

## Out of scope

- User-scoped Connect.
- Automatically wiring addmein's Vercel + Neon connectors.
- Any change to eve itself (it already provides `connect()`).
